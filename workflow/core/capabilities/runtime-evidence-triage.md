# Capability: runtime-evidence-triage

- **Tier**: recommended
- **Stage**: `/03`, `/05`, `/07`
- **Purpose**: 用运行态证据替代静态猜测，排查部署、路由、配置、日志、消息和定时任务等问题。

## 为什么需要

代码静态阅读只能说明“理论上会怎样”，不能证明目标环境实际加载了什么配置、路由、版本或数据。运行态排查必须记录证据链，并明确每条证据来自哪个环境、哪个时间点、哪个入口。

## 输入

- 目标环境、入口、请求样例或任务触发方式
- 可访问的日志、健康检查、配置快照、管理台或监控结果
- 本地代码与当前分支 / 生产基线说明
- 用户手动提供的运行结果

## 输出

```yaml
result: PASS | WARN | BLOCK
evidence_chain:
  - source: "<日志 / 探针 / 配置 / 请求 / 管理台>"
    environment: "<环境>"
    time: "<时间>"
    observation: "<观察结果>"
    conclusion: "<能证明什么，不能证明什么>"
gaps:
  - "<缺少的证据>"
next_checks:
  - "<下一步只读核查建议>"
```

## 阻断规则

- 缺少目标环境证据时，不得把本地代码结论写成线上事实。
- 涉及线上问题时，如未由用户手动刷新或提供生产基线，必须标记“未经远端刷新确认”。
- 运行态证据互相冲突时，阻断“已定位根因”结论，必须先补证据。

## Adapter 示例

- **L0**: 要求排查记录必须包含环境、时间、入口和证据来源。
- **L1**: prompt 按日志、配置、版本、路由、数据五类索要证据。
- **L2**: slash command 生成排查表。
- **L3**: 本地 validator 检查报告中是否存在证据链字段。
- **L4**: subagent 读取用户提供结果并整理事实 / 推断 / 缺口。

## 反模式

- 只看代码就断言线上行为。
- 只有报错截图，没有请求、时间和环境。
- 把配置文件存在等同于目标环境已加载。
- 没有区分“请求到达应用但路由不匹配”和“请求根本没到应用”。
