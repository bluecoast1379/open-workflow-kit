# Checklist: data-consistency-review（数据一致性审查）

适用：金额/数量统计、时间范围筛选、数据导出、手工 SQL、数据补录与订正。

## A. 统计口径

- [ ] 金额/订单数/量级统计必须显式区分**业务主状态字段**与**辅助标记字段**；禁止用辅助标记（是否退款、是否删除、是否作废）反推业务成立（"未退款"≠"已支付"）。
- [ ] 统计口径分散在多个 SQL 分支/子查询时，先列全所有分支，再逐一对齐同一主状态条件；一处改对、其余漏改会造成静默数据偏差。

## B. 时间列口径

- [ ] 按时间筛选/计费/开窗前，先确认业务时间口径是哪一个：创建、开始、完成、结算、财务出账各不相同。
- [ ] 核对四者一致：DTO 字段 → Mapper/查询列 → DB 列 → 线上实际命中量；选定字段处写行内注释钉死口径。
- [ ] 对"上游非必填、平台不回填"的时间列，评估其恒 NULL 时筛选是否塌陷（命中 0 条或全量）；改动前全域 grep 该列的写入点，确认它真的有数据。

## C. 历史数据兼容

- [ ] 长生命周期表的枚举/字典字段解析，必须在转换方法内集中兼容三类输入：历史码值、名称、外部接口值；转换失败返回明确业务异常，禁止 null 穿透到下游解引用或对外接口。
- [ ] 单条脏数据（空字段、未知码值）不得拖垮整批处理（导出中断、缓存重建失败）；转换层逐字段兜底。

## D. 手工 SQL 与数据补录

- [ ] 绕过实体层的手工 SQL 写库，除核对 DB 列类型外必须核对**实体映射类型与格式约定**——写入不符类型的值会"写入成功、读取时炸"；影响面按该表加载方式（是否整表缓存重建）评估。
- [ ] 用 MAX+1 动态分配主键补数时，同一父级新增 ≥2 条必须防同批自撞：一条 `INSERT...SELECT` 里多行看同一快照会分到相同主键；拆逐条 INSERT 或显式加组内偏移，并配 NOT EXISTS 保幂等。
- [ ] 补数/订正脚本出手前必须在预演库真跑一遍，不能只靠 code review。
- [ ] 批量 DML 前，用与 DML **相同 WHERE 的 COUNT** 钉死真实影响行数；工单页"扫描/影响行数"是估算，不可作依据。临时改状态必须登记"恢复条件 + 执行顺序"。

## E. 导出与列表一致性（导出三对齐）

- [ ] 导出 WHERE 条件集 ⊇ 列表 WHERE 条件集（筛选不生效会导全量）。
- [ ] 勾选导出的主键字段必须在列表**真实返回结构**里存在——判断"某结构含某字段"看接口真实返回类型 + SQL SELECT 列，不看同名 VO（同名 id 可能是不同表主键）。
- [ ] 导出 DTO 字段名与前端提交字段名一致；导出条数用勾选数校验而非总数；无业务唯一键的 VO 禁止用作去重键（会静默丢行）。

## F. 敏感值边界

- [ ] 脱敏后的展示值（掩码手机号、掩码卡号等）**禁止**作为等值查询条件或发往第三方——查询命中恒空或对端校验失败；查询与外发只能用原值，取原值路径要有权限控制。
