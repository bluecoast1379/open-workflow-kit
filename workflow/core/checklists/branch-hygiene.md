# Checklist: branch-hygiene（分支卫生与发布搬运）

适用：功能分支管理、发布候选核查、被污染分支补救、合并冲突处理、共享制品版本管理。与 `branch-gatekeeper`、`release-safety-checker` 联动：能力负责阻断判定，本清单给出逐项操作口径。

## A. 分支流向（只出不进）

- [ ] 功能分支只允许**流出**：`功能分支 → 集成分支`、`功能分支 → 生产分支`；**永不** `集成分支 → 功能分支` 反向合并。
- [ ] 机理：集成分支是所有人开发分支的汇集地，内容是大杂烩；任何反向合并都会把他人需求代码与测试环境配置夹带进功能分支，若整分支上线即是发布事故。
- [ ] 需要同步他人基线时：等该需求上生产后从生产分支合，或从该需求源分支单独合；绝不从集成分支合。
- [ ] 被反向合并污染过的分支：锁定为历史分支，禁止继续作为开发、提测、上线、回滚或构建来源。

## B. 发布候选溯源

- [ ] 生产发布候选必须从生产基线创建并可追溯（祖先校验通过）；分支名、测试通过、已合入集成分支都不能替代基线核查。
- [ ] 上线前强制执行并留痕：`生产基线..发布候选` 的提交数、`--first-parent` 提交清单、`--name-status` 文件清单、高风险文件扫描；净变更文件逐个确认属于本期。
- [ ] 差异超出本期登记范围即默认按 P0 阻断，禁止用"测试环境已通过"或"可能不会整分支上线"降级。
- [ ] 未经用户手动刷新远端引用时，结论必须标注"基于本地缓存引用，未经远端刷新确认"。

## C. 共享制品版本

- [ ] 版本号两头防：既不能把开发快照版本（如 `*-SNAPSHOT` 测试号）留进发布候选，也**不能复用已发布的公共制品版本号承载本次新改动**——同号两套内容会覆盖已发布制品、污染所有依赖方；改了公共制品就 bump 新版本号并让全下游统一改引用。
- [ ] 警惕本地同号覆盖构建制造的"编译通过"幸存者偏差：本地能过 ≠ 制品仓库/CI/生产能过。
- [ ] 多需求并行共享公共制品时：功能分支可用隔离版本，合入集成/生产分支时版本一律归一，禁止把开发隔离版本残留进集成分支。
- [ ] 契约变更的发布顺序：公共制品先入库 → 提供方先部署暴露新接口 → 调用方后部署启用；顺序错会导致调用方运行期方法不存在/404。
- [ ] 改共享包的 VO/DTO/契约后，列出**所有**下游消费方（含零代码改动、仅反序列化新字段的调用方）随新版本重建，多仓按依赖顺序一次性构建，禁止乱序发布子模块。

## D. 被污染分支补救（白名单重建法）

- [ ] 确认污染后，基于最新生产基线**重建干净分支**，只搬本需求白名单文件；不在污染分支上原地"洗文件"（历史仍留夹带提交，易漏、不干净）。
- [ ] 共享文件（生产侧也改过的）用 `git merge-tree` 结果搬运，避免整文件搬运回退生产已上线改动。
- [ ] 同文件"本需求 + 夹带"混改的，用 hunk 级 patch 只搬本需求改动，绝不整文件搬。
- [ ] 搬运后四项自核缺一不可：①白名单 grep 无夹带 ②黑名单 grep 无本需求遗漏 ③文件/提交数量吻合 ④依赖闭合能真实编译。grep 特征必须覆盖**所有已识别的夹带类别**，特征不全会假阴性。
- [ ] 自核之外必须安排**独立第三方复核**——自己分类 + 自己 grep 的组合会系统性漏检。

## E. 合并卫生与回归守护

- [ ] 合并/变基冲突解决属于开发行为：解决后先显式保存文件，再用 `git diff` 核对磁盘最终内容；不得把编辑器未保存内容当成已提交事实。
- [ ] 声称"diff 干净"必须跑 `git diff --check`（覆盖 CRLF 新增行的隐藏回车符）；无强制 CRLF 的仓库新增行统一 LF。
- [ ] **回归恢复守护**：凡某文件历史上发生过"已上线修复被 merge 静默抹掉"，登记该修复的特征字符串清单（关键 setter、过滤条件、返回语句）；此后每次涉及该文件的合并，冲突解决后逐一复核特征串。合并的危险在"无冲突部分被静默合错"，只看冲突标记不够。
- [ ] 关键业务口径用行内注释钉死在代码旁（与代码同生共死）；此类修复须安排**生产首验用例**——集成分支可能长期是修复态而生产是回归态，测试通过不能证明生产正确。
