# Checklist: validation-change-review（校验口径变更审查）

适用：任何"放开/弱化必填或强校验"或"收紧字段口径/新增不变式"的改动。此类改动的缺陷在编译期与常规静态审查中都不可见，必须在实现阶段（而非联调阶段）按本清单复扫。

## A. 放开校验方向 —— 空值复扫五步法

- [ ] **VCR-01 跨层盘点全部校验点**：同一"必填/强校验"常在多层重复——前端校验、批量导入行校验器、入口 DTO 注解（如 `@NotNull`）、service 业务校验、对外契约层校验。逐层列出清单并逐层放开；只改一层必漏。
- [ ] **VCR-02 建立解引用清单**：对被放开字段**及其关联对象**（放开 A 常连带 A 的派生对象），在受影响的整方法体、方法内调用的私有方法、同一执行路径（含消息队列、定时任务、结算等异步落库后链路）上，grep 出**全部**解引用点。
- [ ] **VCR-03 逐条 null 化**：每个解引用点要么加判空守卫并给兜底取值，要么确认被互斥门控隔离，要么确认是死代码。注意逻辑运算符**左操作数先求值**——`a.compareTo(b) != 0 && flag` 即使有 flag 也会先解引用 a/b，判空/门控必须写在左侧。
- [ ] **VCR-04 覆盖校验之前的前置依赖调用**：校验语句之前的远程查询/契约调用（对空入参会直接抛业务异常的）也要纳入清单——校验放开了，更早的查询仍会因缺数据抛错。
- [ ] **VCR-05 确认落库口径**：被放开字段若对应 NOT NULL 列，确认空值落库策略（ORM 省略 null 时的 DB 默认值），并实测无约束报错。

### 机理提示（为什么必须写代码时做完）

- **多个空指针会沿执行路径依次掩盖**：修好一个才暴露下一个，靠联调逐个发现需要反复重新部署，代价极高。
- **解析前置拦截链路兜不住**：批量导入类链路常在解析阶段先行校验，失败行直接进失败缓存、根本不进业务层——只放开业务层，导入链路仍然失败。

## B. 收紧口径方向 —— 全写入路径覆盖

- [ ] **VCR-06** 改变字段口径或新增不变式时，grep 出**所有**写入点（新建、更新、批量导入、异步回写）逐一覆盖；只加在 UPDATE 路径会漏 CREATE 路径。
- [ ] **VCR-07** 明确 DML 回补只能修存量，挡不住新增写入路径继续产生坏值；两者都要做。

## C. 验收要求

- [ ] **VCR-08** 每条被放开/收紧的链路**分别实测**：对外接口链路 ≠ 批量导入链路 ≠ 页面提交链路，一条链路 5/5 通过不代表其他链路通过。
- [ ] **VCR-09** 在 `/05-代码审查` 与 `/07-测试执行` 把"全部校验点是否逐层处理 + 解引用是否逐条判空 + 每条链路是否实测"列为专项检查项。
