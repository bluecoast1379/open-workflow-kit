# Checklist: third-party-integration-review（第三方集成审查）

适用：第三方接口调用、异步回调、消息消费、数据变更扇出通知。本清单是 `protocol-state-machine-checker` 与 `contract-tracer` 的逐项执行版。

## A. 事实源与对账

- [ ] **TIR-01** 本地状态/失败原因字段只反映"最后一次调用结果"，只能作展示线索；**订正或对账必须以第三方权威台账为准**，大规模订正前先取对方导出对账。
- [ ] **TIR-02** "我方调用成功"仅代表发出成功，业务结果以第三方账为准。

## B. 多步协议状态机

- [ ] **TIR-03** 多步协议在架构阶段画状态机：每步前置条件、"成功"指哪一步走完、可重入性；引入"已开始未完成"中间态，禁止用二元成败字段表达多步协议。
- [ ] **TIR-04** HTTP 200 ≠ 协议成功；逐步核对业务码。
- [ ] **TIR-05** 响应判定顺序：**幂等成功文案（已存在/已上传/请勿重复）必须先于失败关键字识别**，否则幂等成功被误判为失败反复重试。

## C. 失败处理

- [ ] **TIR-06** 第三方调用失败必须显性化为异常或失败终态；禁止"吞异常 + 返回空对象"式静默兜底——它把"第三方失败"伪装成"成功但字段空"，写入脏状态。
- [ ] **TIR-07** 关键状态写入前，强校验第三方返回的关键标识非空。
- [ ] **TIR-08** 自动重试先分类：**可重试**（超时/网络/5xx）与**永久失败**（业务规则拒收）；永久失败不重试，否则确定性失败被无限重推形成调用风暴。
- [ ] **TIR-09** 同步改异步必须配套：超时兜底、定时补捞、失败可筛可告警、人工处理 SOP——"转人工"是需要触达闭环的起点，不是终点。

## D. 变更扇出与消息消费

- [ ] **TIR-10** 由数据变更（CDC/binlog/触发器）驱动的对外通知，底层是"一主多明细/多行写库"时，必须在**业务事件粒度**聚合/节流/幂等，禁止行级直通（一次业务操作被放大成 N 次通知）。
- [ ] **TIR-11** 节流/去重/合并改动必须显式证明"不丢最新"不变式；尾触发补偿用持久化机制（延迟消息/落库定时），不用进程内定时器（重启即丢）。
- [ ] **TIR-12** 严格顺序消费的消费方法**绝不向消息中间件抛异常**：内部兜底（失败转重试、落库）自身也要被捕获，落库失败只能告警 + 确认消费，异常逃逸会阻塞整条队列。

## E. 运行态排查顺序

- [ ] **TIR-13** 接口 404/异常先证伪代码与路由（应用内框架格式的 404 说明请求已达应用，优先查路径拼接/上下文前缀叠加），再怀疑构建部署缓存；避免反复"白重构建"。
- [ ] **TIR-14** "构建成功 ≠ 启动成功 ≠ 部署生效 ≠ 路由可达"：发布后确认新实例就绪、启动日志无失败、代表性接口探活可达（详见 deployment-readiness-checker）。
