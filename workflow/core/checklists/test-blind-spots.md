# Checklist: test-blind-spots（测试盲区清单）

适用：`/06-测试用例` 设计与 `/05-代码审查` 独立复审。以下 10 类盲区来自真实复盘的共同模式："测试用例全部通过，但缺陷仍然漏到独立复审或生产"。

## A. 十类高频盲区（用例设计必须逐类核对）

- [ ] **1. 负向/中间状态**：只测"成功/正常/已支付"，漏掉"退款中、未完成、作废、失败、免征"等应被排除或纳入的状态；每个状态字段列全枚举再决定纳入/排除。
- [ ] **2. 兼容入口矩阵**：同一业务规则的多条入口（对外接口、批量导入解析、页面提交）只测主入口；导入解析常先于业务层校验，主入口通过不代表导入链路通过——每条入口分别设计用例。
- [ ] **3. 删除操作副作用**：只测新增/修改，漏删除引发的额度回收、绑定解除、缓存清理、空集合仍对外调用等副作用。
- [ ] **4. 权限/租户隔离**：漏跨租户数据可见、越权操作；注意"串数据"错觉可能源于快照口径而非筛选缺陷，改错方向反而制造真隔离事故。
- [ ] **5. 重复请求/幂等**：漏并发重复提交、先查后插竞态窗口、消息重投；前端防抖只是体验优化，后端必须有唯一键或锁兜底。
- [ ] **6. 中间态/混合态**：单一状态位无法表达"已开始未完成"；父子/批次状态错位需要构造数据才能覆盖——用例的"场景描述"必须与"实际数据形态"对齐。
- [ ] **7. 历史脏数据/未知码值**：单条空字段或未知枚举拖垮整批（导出中断、缓存重建失败）；用历史形态数据实测。
- [ ] **8. 大数据量与并发压力**：小数据量通过≠安全；按"大批量 + 高并发 + 第三方故障"三种压力推演：全量装载内存、边处理边翻页塌缩漏单、LIMIT+后置过滤饥饿。
- [ ] **9. 死字段/未落地**：字段/分支"看似实现"实为占位（零写入、恒 0、恒空、恒 false 保护）——编译通过但功能为空。
- [ ] **10. 环境差异**：测试环境同构、白名单、配置注入、网关收口差异会掩盖问题；识别"测试环境无法复现"的项并显式标注"上线后必验"，安排生产首验用例。

## B. 独立对抗式复审（关键路径必做）

适用：金额、契约、性能重构、共享组件、口径变更类改动。除作者自审外必须引入一轮**不带自审结论**的独立复审，且必须包含"需求达成审查"：

- [ ] **逐条落点**：产品文档需求项逐条指到具体代码落点；指不到即未实现。
- [ ] **死字段检测**：对新增字段/新列 grep 其 setter 与写入点，零写入 = 未实现缺陷，不是"待测试"。
- [ ] **占位即缺陷**：占位值、恒 0、TODO、"由他处写入"的注释声明，必须找到真实写入点，否则按未实现缺陷计，不得用"待测试"稀释级别。
- [ ] **注释必验**：注释声明的行为必须用代码验证，注释不是证据。
- [ ] **端到端追踪**：关键链路从入口逐段追到终点（查询条件/落库/外发），不停在中间层。
