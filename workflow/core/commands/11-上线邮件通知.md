# /11-上线邮件通知

## Goal

上线通知: 形成上线范围、影响、时间、回滚口径和注意事项。

## Required Inputs

- `AGENTS.md`
- `workflow/team-profile.yaml`
- Previous stage documents under workspace-level `features/{feature}/`
- `features/{feature}/completion/contract.yaml`（Completion Contract）、Evidence Ledger 与 release candidate
- Local code, local docs, and user-provided source materials listed in team-profile

## Execution Rules

- Read local facts before writing conclusions.
- Distinguish verified facts, design intent, assumptions, and missing evidence.
- 默认使用简体中文展示工作流沟通和阶段产物；专有名词、产品名、品牌名、代码标识符、命令、文件路径、分支名、API、SDK、框架、协议、标准、错误信息和官方英文术语保留原文。
- Do not claim tests, builds, screenshots, deployments, or reviews passed unless they were actually executed.
- Git、数据库、部署、生产配置和外部写操作统一按 `workflow/core/execution-policy.md` 四层最严格结果执行；通知文档不等于发布授权。
- This stage does not authorize business code changes unless the current command is an implementation command and all gates pass.

## Required Structure

- 受众、发布时间、release source / target（来自 team-profile）、release commit、范围 / non-goals。
- 用户与业务影响、迁移 / 兼容 / 停机、质量与验收状态、已知风险和有效 waiver。
- 部署方式与有效授权、CI / Evidence Ledger、secret / config / data 变更、监控 / 告警 / runbook。
- 灰度、kill switch、回滚 / 恢复、RTO/RPO、负责人、升级渠道与上线后 North Star 观察窗。

## Exit Criteria

- 通知内容与当前 contract/source/release evidence 一致，无旧 commit 绿灯或分支名假设。
- 发布、回滚、监控、数据 / 配置、用户沟通 Owner 和时间窗明确。
- `RELEASED` 只在真实发布后使用；发布前通知明确标记计划，文档不被当成执行授权。
- 安全隐私敏感信息已脱敏，known issues / waiver / fallback 不被省略。



## Required Outputs

- Update or create the corresponding file under workspace-level `features/{feature}/`.
- Update workspace-level `features/{feature}/00-工作流状态.md` when stage status changes.
- Record unresolved questions and evidence gaps explicitly.
- Record release source and target from `workflow/team-profile.yaml#branch_model` (never assume branch names), CI evidence, deployment mode, effective approvals, secret/config changes, rollback plan, and whether automation is disabled, manual-gated, or guarded-auto.
