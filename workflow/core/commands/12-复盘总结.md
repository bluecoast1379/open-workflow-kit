# /12-复盘总结

## Goal

复盘的不只是实现和缺陷，还要衡量“定义是否足够清楚、证据是否可信、自主循环是否有效、上线后价值假设是否成立”，并形成可复用但已脱敏的改进候选。

## Required Inputs

- `features/{feature}/completion/contract.yaml`（Completion Contract）全版本、Evidence Ledger、run-state 与 01–11 阶段产物
- Git / CI / 测试 / 发布 / 运行态证据和事故记录
- 人工验收反馈、North Star 与 guardrail 的真实观测（尚未到观察窗则明确 PENDING）
- 决策、假设、豁免、scope drift 与 Agent 自主循环历史

## Execution Rules

- 以事实和时间线为基础；不把相关性写成因果，不把“上线”写成“商业成功”。
- 区分定义缺陷、实现缺陷、Oracle 缺陷、证据缺陷、工具 / 环境缺陷和组织决策延迟。
- 私有事实先脱敏；只有跨项目稳定、无敏感内容且有证据的模式才进入 rule / capability 候选。
- 本阶段不自动修改 core 规则、业务代码或外部系统；改进项另行评审与授权。

## Required Structure

1. **结果快照**：最终 contract/source/release、AC 通过率、human acceptance、上线与回滚状态。
2. **业务结果**：North Star / guardrail 的 baseline、target、实际值、观察窗、归因限制、继续 / 调整 / 停止决策。
3. **定义质量**：冻结后需求 / AC 变更、首次人工验收通过率、歧义、遗漏的 non-goal / invariant / 异常 / NFR / human gate。
4. **证据质量**：覆盖率、新鲜度、不可复现 / flaky、错误 Oracle、mutation / negative control 表现和逃逸缺陷。
5. **自主循环效率**：迭代数、耗时、人类注意力分钟、token / 模型 / 基础设施成本、重复失败指纹、阻断和预算终止。
6. **工程与运营结果**：性能成本、可靠性、安全隐私、可观测性、回滚 / 恢复、兼容迁移、AI 质量与供应链事件。
7. **组织协作**：DRI / 决策 / 依赖 SLA、审批、豁免、handoff 和共识边界的实际表现。
8. **根因与反事实**：发生了什么、为什么现有 gate 未捕获、哪条最小规则能预防；避免“加强测试”式空话。
9. **改进项**：Owner、优先级、截止、成功指标、影响的 command/capability/policy/schema；区分项目动作与 core 候选。
10. **生命周期**：继续运营、feature flag / 清理、迁移、数据保留删除、依赖移除和最终下线条件。

## Exit Criteria

- 所有关键结论均有证据或明确标为假设；商业观察窗未结束时状态为 PENDING。
- 至少回答定义、实现、验证、证据、Agent 循环、组织和业务价值七类“哪里失真、如何防复发”。
- 改进项具体到 Owner、期限、验证指标和落点；没有把项目特例直接提升为全局规则。
- 可复用候选已脱敏、去重、给出适用边界和反例；需要改 core 时进入单独变更流程。
- 状态、遗留风险、下线 / 运营动作和后续效果验证 Owner 已更新。

## Required Outputs

- `features/{feature}/12-复盘总结.md`
- 更新 `features/{feature}/00-工作流状态.md`
- 可选：脱敏后的 memory / rule / capability / policy-pack 候选清单（仅候选，不自动合入 core）
