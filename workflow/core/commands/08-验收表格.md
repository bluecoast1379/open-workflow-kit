# /08-验收表格

## Goal

从 Completion Contract 与 Evidence Ledger 生成业务可读、逐项可追溯的验收表；它是当前事实的投影，不是人工重新抄写的一套标准。

## Required Inputs

- 冻结的 `features/{feature}/completion/contract.yaml`（Completion Contract）
- 当前 source / environment / findings 对应的 `completion/evidence/ledger.jsonl`、`run-state.yaml`、`environment.yaml` 与 `findings.yaml`
- 01–07 阶段文档、设计证据、风险与豁免记录
- team-profile 中的 Owner、分支、发布和质量预算

## Execution Rules

- 验收表只能引用现有 `AC-###`，不得新增、更名、合并或降低标准；需要变更先走 contract version change。
- Evidence Ledger 是自动结果事实源；手工表格不得覆盖 `FAIL/BLOCKED/NOT_RUN/STALE`。
- `WAIVED` 独立展示，不算 PASS；人工 gate 只显示待签收，不预填通过。
- 本阶段不授权代码、数据库、发布或生产写操作。

## Required Structure

1. **验收快照**：feature、contract version/hash、source/environment/findings fingerprint、ledger head/count、生成时间和生成者。
2. **业务目标**：North Star、baseline、target、观测窗口、guardrail；区分交付验收和上线后效果验证。
3. **AC 明细**：ID、业务描述、关联 REQ/RISK、blocking、Oracle、当前状态、证据链接、证据新鲜度、Owner、human gate。
4. **质量预算**：UX/accessibility、performance/cost、reliability/resilience、security/privacy/compliance、observability、reversibility/evolution、AI quality。
5. **异常与恢复演示**：失败、重试、降级、回滚、数据修复和人工运营入口的验收步骤。
6. **豁免与残余风险**：批准人、范围、理由、到期日、补偿控制和复验计划。
7. **组织签收**：业务、产品、设计、技术、安全 / 合规等实际需要的签收角色；不需要的角色说明 N/A。
8. **最终聚合**：runner 自动状态、findings validity、待人工 gate、Owner-signed ledger anchor 状态、阻断项和允许的下一动作。

## Exit Criteria

- Contract 中每个 AC 恰好在验收表出现一次，ID、阈值、blocking 与 human gate 未改变。
- 每个自动 PASS 有当前 contract/source/environment 的有效证据；过期或缺失证据正确显示为 `STALE/NOT_RUN`。
- 所有 WAIVED 和残余风险透明展示且未过期；未通过项不会被摘要语言掩盖。
- 业务 / UX 人工验收提供具体步骤与 rubric，签收人不需要阅读测试日志即可复核核心结果。
- 自动部分与 runner run-state 一致；人工证据和最终 `ACCEPTED` 以独立 evaluator 为准。此阶段通常显示 anchor 待签，不得把缺失 anchor 写成“基本通过”。

## Required Outputs

- `features/{feature}/08-验收表格.md`
- 更新 `features/{feature}/00-工作流状态.md`；不得修改 Evidence Ledger 历史事实。
