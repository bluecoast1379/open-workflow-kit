# /03-技术架构

## Goal

把 PRD 和 draft Completion Contract 转化为可实现、可验证、可回滚、可演进的技术契约；本阶段细化范围、质量预算和验证 Oracle，但最终冻结由后续 `/define-done` 统一完成。

## Required Inputs

- `AGENTS.md`、`workflow/team-profile.yaml`
- `01-需求讨论.md`、`02-产品文档.md`；涉及 UI 时读取 `02B-UI设计.md`
- `/new-feature` 生成的 draft `features/{feature}/completion/contract.yaml`
- 真实仓库、分支、依赖、运行环境、历史数据和工具链证据

## Execution Rules

- 分支模型、基线与发布流只从 `team-profile#branch_model` 读取，不假设 `main`、`prod` 或 `test`。
- 从入口追踪到持久化、异步、第三方和用户可见终点；不能用 DTO 或接口声明替代真实调用链。
- 方案必须在最坏负载、弱网、超时、重试、并发、内存 / 连接耗尽、部分故障和恢复条件下推演。
- 本阶段仅授权分析与工作区文档；不创建“已实现”“已审查”或“已测试”的虚假产物。

## Required Structure

1. **基线与影响面**：项目族、仓库、branch / commit / dirty 状态、worktree、受影响路径、依赖、数据、配置、协议和发布单元。
2. **架构与数据流**：上下文 / 组件图、入口到终点调用链、实体生命周期、权威数据源、读写归属、事务边界、一致性模型、缓存与失效、side effects。
3. **状态与不变量**：状态机、合法 / 非法迁移、幂等键、去重窗口、原子操作、并发冲突、时间 / 顺序语义和必须保持的不变量。
4. **接口与演进**：API / event / schema / serialization 契约、错误语义、版本兼容、迁移顺序、provider / consumer 发布顺序、废弃与下线。
5. **异常与韧性**：超时、重试 / 退避 / 抖动、熔断、限流、背压、离线、依赖降级、部分成功、补偿、故障注入、RTO/RPO 和爆炸半径。
6. **性能与成本预算**：负载模型、p50/p95/p99、吞吐、容量、内存 / CPU / 电量 / 网络、第三方 / 基础设施 / LLM 单次及月度成本，附测量环境与回归阈值。
7. **安全、隐私与合规**：威胁边界、身份 / 权限、最小权限、秘密、数据最小化、保留 / 删除、加密、审计、恶意输入、滥用路径和适用法规。
8. **可观测性与可运营性**：结构化日志、指标、trace、correlation ID、SLO / error budget、告警、dashboard、runbook、人工修复入口和故障检测目标。
9. **可逆性与发布**：feature flag、灰度、kill switch、数据备份 / 修复、回滚点、向前修复、迁移可逆性和回滚演练。
10. **AI 系统（如适用）**：模型 / prompt / 检索 / 工具版本、确定性边界、评测集、质量分布、安全拒绝、幻觉、回退、缓存和成本控制。
11. **验证架构**：每个 `AC-###` 的 Oracle、层级（unit / integration / contract / e2e / performance / security / manual）、fixture / seed、环境指纹、setup / cleanup、反事实与 mutation / fault injection 候选。
12. **组织与供应链**：组件 Owner、外部 SLA、许可证、版本锁定、SBOM / 替代方案、vendor lock-in、审批与升级路径。
13. **实施计划与闸门**：按依赖排序的变更、允许 / 禁止路径、分支与 worktree 方案、CI 必跑检查、部署目标、secret boundary、回滚假设和自主执行预算。
14. **风险登记**：`RISK-###`、概率、影响、探测、缓解、Owner、关联 AC；未知项与需要用户决策的 decision packet。

## Exit Criteria

- 每个 blocking `AC-###` 有可执行或明确的人工 Oracle、环境、数据、阈值和证据要求；不能验证的条件标为阻断而非“后续补”。
- 需求 → 架构组件 / 数据流 → 实现任务 → 测试 Oracle 可双向追溯。
- 所有关键写入、状态迁移、外部协议和多入口规则的原子性、幂等、失败与恢复语义完整。
- 性能、成本、韧性、安全隐私、可观测性、可逆性、演进和 AI 质量预算均可验证或有明确 N/A 理由。
- 真实分支 / worktree / 基线与 `team-profile` 一致；实现范围、并行隔离和发布顺序无歧义。
- 所有 P0/P1 风险已缓解、由 AC 覆盖或形成待用户决策的阻断包。
- Completion Contract 的技术字段、Oracle 与预算已细化到可供 `/06-测试用例` 和 `/define-done` 复核；本阶段不得把 draft 自行标记为 Owner 已冻结。
- 实现准入结论只能是 `PASS` 或 `BLOCKED`；不以文档完整度替代真实基线检查。

## Required Outputs

- `features/{feature}/03-技术架构.md`
- 细化 `features/{feature}/completion/contract.yaml` draft；不得代替 `/define-done` 执行最终冻结。
- 更新 `features/{feature}/00-工作流状态.md`，记录实现准入、分支 / worktree 与阻断项。
