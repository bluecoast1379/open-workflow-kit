# /01-需求讨论

## Goal

把模糊需求转化为可审计的问题定义、价值假设、范围边界和初始验收条件；先定义“为什么做、为谁做、什么绝不能变”，再讨论方案。

## Required Inputs

- `AGENTS.md`、`workflow/team-profile.yaml`
- 用户原始需求、业务资料、研究证据、现有产品 / 代码事实
- `features/{feature}/00-工作流状态.md` 与已有 Completion Contract（续作时）

## Execution Rules

- 先查证本地事实；明确区分 `FACT`、`ASSUMPTION`、`DECISION` 与缺失证据。
- 默认简体中文展示，技术标识与官方术语保留原文。
- 不用“高性能”“体验流畅”“足够安全”“智能”等形容词代替可验证阈值。
- Agent 可以提出候选验收条件，不能替业务 Owner 决定商业目标或替体验 Owner 签收审美判断。
- 本阶段仅授权分析和工作区级文档，不授权修改业务代码或执行外部写操作。

## Required Structure

阶段文档必须包含：

1. **问题与目标**：当前状态、目标状态、目标用户、使用场景、触发条件、为什么现在做。
2. **商业价值**：North Star、baseline、target、观测窗口、guardrail、停止条件；无法量化时写出判断 rubric 和 Owner。
3. **范围边界**：In scope、Non-goals、反需求、禁止顺手重构项、必须保持的不变量。
4. **业务与数据语义**：实体、术语表、权威数据源、关键状态、输入 / 输出、前置 / 后置条件；歧义词给出正例、反例和边界例。
5. **用户与体验**：主任务、用户动机、认知负荷、犹豫点、误操作、恢复路径、信任风险、无障碍和反暗黑模式约束。
6. **组织共识**：DRI、决策者、评审者、依赖 Owner、响应 SLA、分歧与升级路径。
7. **假设与不确定性**：`ASM-###`、证据、置信度、验证方法、Owner、截止 / 失效时间；未知未知的探测策略。
8. **初始风险**：技术物理边界、第三方、数据、安全隐私、合规、性能成本、可运营性、可逆性与 AI 非确定性（如适用）。
9. **初始验收条件**：稳定 `AC-###`、优先级、blocking、Given / When / Then、候选 Oracle、所需证据、human gate；每条 AC 关联目标或风险。
10. **待确认项与决策包**：选项、权衡、推荐项、最晚决策时间；不得用开放问题掩盖阻断项。

## Exit Criteria

- 问题、目标用户、业务目标、In scope、Non-goals 和不变量均有明确文字，彼此不矛盾。
- North Star 或人工商业判断 rubric 有 Owner、baseline / target（如适用）、观测窗口与 guardrail。
- 每个 P0/P1 用户结果和关键风险至少有一个初始 `AC-###`；体验型与商业型条件明确标为 automated 或 human gate。
- 所有模糊术语已被阈值、决策表、示例或 rubric 消融；无法消融者记录 `ASM-###` 或阻断决策。
- 组织 Owner 和外部依赖边界完整；阻断性未知项不得带入实现准入。
- 文档、`00-工作流状态.md` 和 Completion Contract 草案一致；本阶段没有虚报执行证据。

## Required Outputs

- `features/{feature}/01-需求讨论.md`
- 创建或更新 `features/{feature}/completion/contract.yaml` 的 objective、outcome、scope、glossary、assumptions 与 acceptance 草案。
- 更新 `features/{feature}/00-工作流状态.md`。
