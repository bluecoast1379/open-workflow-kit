# /07-测试执行

## Goal

在受控环境真实执行与 `AC-###` 绑定的 Oracle，生成可复核、可失效的 Evidence Ledger，并诚实区分通过、失败、阻断、未运行、过期和豁免。

## Required Inputs

- `AGENTS.md`、`workflow/team-profile.yaml`
- 冻结的 `features/{feature}/completion/contract.yaml`（Completion Contract）、`06-测试用例.md` 与可执行测试计划
- 当前 source / environment / dataset / tool fingerprint
- 已完成的 04 实现与 05 审查证据（适用时）

## Execution Rules

- 每次执行前记录 contract hash、source fingerprint、environment fingerprint、fixture / seed 和工具版本；任一不一致时旧证据先标 `STALE`。
- 按 `automated-test-runner` 双轨和相关能力执行；目标环境必须在 allowlist，凭证只从未跟踪的本地凭证文件读取并脱敏。
- `/07` 本身不授权修改业务代码。失败项回到 `/04-代码实现`，或由已获实现授权的 `/deliver-until-done` 循环修复后重新进入 `/07`。
- 禁止删除失败用例、降低阈值、用重跑隐藏 flaky、把部分测试 / 构建 / 页面打开当完整 AC PASS。
- 所有高风险动作按 `execution-policy`；自动测试不替代 `/09` 的人工签收。

## Required Structure

1. **执行上下文**：contract version/hash、branch/commit/dirty、环境与依赖指纹、数据集 / fixture / seed、工具 / 模型版本、时间和执行者。
2. **AC 结果矩阵**：`AC → TC/Oracle → PASS/FAIL/BLOCKED/NOT_RUN/STALE/WAIVED → evidence`；聚合规则不得用多数票覆盖 blocking 失败。
3. **分层执行结果**：build / lint / unit / integration / contract / e2e / browser / performance / cost / security / accessibility / resilience / AI eval / manual-prep。
4. **证据清单**：命令 hash、退出码、耗时、断言、artifact 路径与 SHA-256、日志 / 截图脱敏状态、supersedes 关系。
5. **失败与 flaky**：失败指纹、复现步骤、根因证据、Owner、回修 AC；flaky 重跑历史和 quarantine 到期日。
6. **运行态与发布候选**：日志 / 指标 / trace / 告警、发布候选与 team-profile 所定义生产基线的差异、路由可达、迁移与回滚演练。
7. **防作弊与反事实**：negative control / mutation / fault injection 结果，确认关键 Oracle 能捕获错误实现。
8. **结论**：只能是 `READY_FOR_HUMAN_ACCEPTANCE`、`BLOCKED_WITH_DECISION_PACKET` 或 `BUDGET_EXHAUSTED`，附未满足项。

## 强制检查清单

- `test-blind-spots.md` §A-10：环境差异和“上线后必验”。
- `data-consistency-review.md` §D：含 DML 时核对 COUNT、预演与恢复证据。
- `branch-hygiene.md` §B：发布候选与 team-profile 生产基线差异。
- `validation-change-review.md` §C：接口、导入、页面、异步链路分别实测。
- `third-party-integration-review.md` §B/§C/§E：业务码、幂等、重试 / 永久失败、异步闭环和路由可达。

## Exit Criteria

- 当前 contract/source/environment 上所有非人工 blocking AC 为 `PASS` 或存在合规且未过期的 `WAIVED`；这些自动 AC 不得存在 `FAIL/BLOCKED/NOT_RUN/STALE`，且当前 findings snapshot 有效、无开放 P0/P1。人工 blocking gate 在 `/09` 有权角色按 rubric 签收前保持 pending/`NOT_RUN`，不阻断 `/07` 进入 `READY_FOR_HUMAN_ACCEPTANCE`，但阻断 `ACCEPTED`。
- 无开放 P0/P1 finding、无 scope drift；性能 / 成本 /可靠性 / 安全隐私 / 可访问性 / AI 等硬预算全部达标。
- 每个 PASS 都有可复现 Oracle 与可校验 artifact；“未发现问题”、静态阅读、计划和旧 commit 证据不算 PASS。
- flaky 未被重跑掩盖；quarantine、外部不可用和无法执行项不会被聚合为通过。
- 关键 Oracle 至少通过 negative control、mutation 或等价反事实证明具有检错能力，或记录明确的残余风险与 human gate。
- Evidence Ledger、测试报告和 `00-工作流状态.md` 聚合一致；满足时只能宣布“自动验收就绪”，不能代替人工 `ACCEPTED`。
