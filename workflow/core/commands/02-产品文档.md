# /02-产品文档

## Goal

把已确认的问题定义编译为可追溯的 PRD：稳定需求 ID、确定性业务规则、用户流程、质量预算与可验证验收条件。详细 UI/UX 基线仍由 `/02B-UI设计` 承接。

## Required Inputs

- `AGENTS.md`、`workflow/team-profile.yaml`
- `features/{feature}/01-需求讨论.md`
- `features/{feature}/completion/contract.yaml`
- 现有产品、代码、数据、设计规范与用户证据

## Execution Rules

- 每项功能需求使用稳定 `REQ-###`；每条 blocking `REQ` 至少映射一个稳定 `AC-###`。
- 业务规则写成决策表、状态转换、公式或前后置条件，不以散文替代确定性。
- 每条 NFR 必须有单位、阈值、负载 / 环境条件和验证方式；无依据的阈值标为假设，不伪装成事实。
- Agent 不得为减少实现量删除用户已确认范围或降低 Completion Contract 阈值。
- 本阶段只授权工作区级文档，不授权业务代码修改。

## Required Structure

1. **文档控制**：版本、Owner、评审者、事实源、关联决策与变更摘要。
2. **目标与指标树**：业务目标 → 产品行为 → 可观测事件 → North Star / guardrail；包含 baseline、target、时间窗和失败止损。
3. **用户与场景**：角色、权限、Jobs-to-be-Done、主流程、异常 / 恢复流程、不可接受体验。
4. **范围**：In scope、Non-goals、向后兼容边界、不变量、公开契约和禁止修改项。
5. **需求清单**：`REQ-###`、优先级、输入、处理、输出、依赖、Owner、关联 `AC-###`。
6. **业务规则与数据逻辑**：实体生命周期、权威数据源、状态机、决策表、并发 / 幂等 / 原子性、时区与顺序语义、部分成功和补偿。
7. **UX 高层方向**：信息架构、反馈时机、空 / 错 / 加载 / 权限 / 离线状态、恢复成本、无障碍、信任与克制约束；需要前端时声明 02B 闸门。
8. **质量预算**：性能 p50/p95/p99、吞吐与容量、资源 / 电量 / 网络、可靠性、RTO/RPO、单次与月度成本、安全隐私、可访问性；不适用项说明理由。
9. **可观测性与运营**：事件口径、日志 / 指标 / trace、告警、人工处理入口、runbook 和数据保留。
10. **演进与可逆性**：feature flag、灰度、kill switch、回滚、数据恢复、API/schema 兼容窗口、迁移与下线策略。
11. **AI 质量（如适用）**：模型 / prompt 版本、黄金集、质量分布、幻觉与安全拒绝指标、方差、回退和推理成本。
12. **验收矩阵**：`AC-###`、关联 `REQ/RISK`、Given / When / Then、Oracle、阈值、环境、数据、证据、新鲜度、blocking、human gate。
13. **依赖与组织边界**：DRI、决策权、外部 Owner、SLA、审批 / 合规和合同变更重签条件。

## Exit Criteria

- 所有 P0/P1 `REQ-###` 均映射到一个或多个 `AC-###`，且没有孤立 AC。
- 业务规则覆盖正常、异常、边界、权限、并发、重试、部分成功与恢复路径；相同规则的多入口被分别定义。
- 每个“快、稳定、便宜、安全、流畅、智能”等描述都已转换为阈值、示例或人工 rubric。
- 性能 / 成本 /可靠性 / 安全隐私 / 可观测性 / 可逆性 / 演进 / AI 质量已定义或有可审计的 N/A 理由。
- 商业指标、埋点事件和观测窗口形成闭环；技术通过与商业成功被明确区分。
- UI/前端工作已声明 `/02B-UI设计` 闸门；PRD 不冒充详细设计。
- Completion Contract 与 PRD 的 REQ/AC、范围和质量预算一致，通过定义质量人工核查；阻断未知项不得进入技术架构准入。

## Required Outputs

- `features/{feature}/02-产品文档.md`
- 更新 `features/{feature}/completion/contract.yaml` 的 requirements、domain、quality_budgets、acceptance 与 organization。
- 更新 `features/{feature}/00-工作流状态.md`。
